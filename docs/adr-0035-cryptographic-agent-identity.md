## ADR-0035: Cryptographic Agent Identity

### Context

Today, agent identity in cagent is implicit. The supervisor spawns workers and
attestors as `cagent run` subprocesses, but there is no cryptographic binding
between an agent's identity and the artifacts it produces. The attestation nonce
(ADR-0026) prevents workers from self-attesting, but a compromised or rogue
agent could forge updates, notes, or commits attributed to another agent. As
cagent moves toward choir (distributed multi-VM orchestration), the trust
boundary expands from a single machine to a fleet of VMs, making cryptographic
identity essential.

### Decision

The supervisor becomes a Certificate Authority (CA). Every agent run receives a
short-lived Ed25519 keypair and a capability token that scopes what the agent
can do. Workers sign their git commits. Attestors sign their attestation
records. The work graph becomes a cryptographically verifiable audit trail.

### Design

#### 1. Key Hierarchy

```
cagent (local)                          choir (distributed)
─────────────────                       ─────────────────────
Supervisor CA                           Hypervisor Root CA
  └─ per-run agent keypair                └─ per-VM intermediate CA
                                              └─ per-agent keypair
```

**cagent (local):** The supervisor generates a long-lived Ed25519 CA keypair on
first run, stored at `$CAGENT_STATE_DIR/ca.key` (private) and `ca.pub`
(public). Each `cagent run` invocation receives an ephemeral Ed25519 keypair
generated by the supervisor at dispatch time. The CA signs the agent's public
key to produce a certificate binding the key to {work_id, job_id, role,
capabilities, expiry}.

**choir (distributed):** The hypervisor holds the root CA key. Each VM receives
an intermediate CA certificate signed by the hypervisor. The VM's agent runtime
generates per-agent keypairs signed by the VM's intermediate CA. This creates a
three-level chain: hypervisor → VM → agent → artifacts. The hypervisor root key
is the only long-lived secret; VM and agent keys are ephemeral.

#### 2. Capability Tokens

A capability token is a signed JSON structure (not JWT — we avoid the JWT
ecosystem's complexity and instead use a minimal Ed25519-signed envelope):

```json
{
  "version": 1,
  "subject": {
    "job_id": "job_01ABC...",
    "work_id": "work_01XYZ...",
    "role": "worker",
    "adapter": "claude",
    "model": "claude-sonnet-4-6"
  },
  "capabilities": [
    "work:update",
    "work:note-add"
  ],
  "issued_at": "2026-03-18T08:30:00Z",
  "expires_at": "2026-03-18T09:00:00Z",
  "issuer_pubkey": "<base64 CA public key>",
  "agent_pubkey": "<base64 agent public key>",
  "signature": "<base64 Ed25519 signature over canonical JSON of above fields>"
}
```

**Role → Capability mapping:**

| Role      | Capabilities                                      |
|-----------|---------------------------------------------------|
| worker    | `work:update`, `work:note-add`                    |
| attestor  | `work:attest` (requires nonce), `work:note-add`   |
| reviewer  | `work:approve`, `work:reject`, `work:note-add`    |
| planner   | `work:create`, `work:edge-add`, `work:note-add`   |

The CLI enforces capabilities: before executing any write command, the CLI
checks the token's capability list. A worker that attempts `cagent work attest`
gets a hard rejection, not just a nonce mismatch.

**Token delivery:** The supervisor writes the token and agent private key to a
temporary file (mode 0600) and passes the path via the `CAGENT_AGENT_TOKEN`
environment variable. The agent runtime reads and deletes the file on startup.
This avoids passing secrets on the command line (visible in `ps`).

#### 3. Signed Git Commits

Each agent run configures git to sign commits with its ephemeral key:

```
GIT_COMMITTER_NAME=cagent-worker-<job_id_short>
GIT_COMMITTER_EMAIL=<job_id>@cagent.local
GIT_SIGNING_KEY=<path to ephemeral SSH signing key>
```

Git supports Ed25519 SSH keys for commit signing natively (`gpg.format = ssh`).
The supervisor writes an `allowed_signers` file containing all active agent
public keys, keyed by their cagent email addresses. After a run completes, the
supervisor can verify commits via `git log --show-signature`.

**Why SSH signing, not GPG:** GPG requires a keyring, agent daemon, and trust
database. SSH signing uses the same Ed25519 keys we already generate, requires
no daemon, and is supported by Git 2.34+. The key material is identical; only
the envelope differs.

#### 4. Signed Attestation Records

The `AttestationRecord` gains two new fields:

```go
type AttestationRecord struct {
    // ... existing fields ...
    SignerPubkey string `json:"signer_pubkey,omitempty"`  // base64 Ed25519
    Signature    string `json:"signature,omitempty"`      // base64 Ed25519 sig
}
```

When an attestor calls `cagent work attest`, the CLI:
1. Validates the capability token (role=attestor, not expired)
2. Validates the nonce (existing behavior)
3. Constructs the attestation record
4. Signs the canonical JSON of the record with the agent's private key
5. Stores the signature and public key alongside the record

Verification checks: (a) the signature is valid for the record content, (b) the
public key is signed by the CA, (c) the CA-signed certificate grants the
`work:attest` capability.

#### 5. Verification Tooling

```
cagent verify <work-id>
```

This command reconstructs and verifies the full signature chain for a work item:

1. **CA trust root:** Load the supervisor CA public key
2. **Agent certificates:** For each job that touched this work item, verify the
   agent certificate was signed by the CA
3. **Commit signatures:** For each commit attributed to the work item, verify
   the git commit signature matches the agent's certified key
4. **Attestation signatures:** For each attestation record, verify the
   signature matches a certified attestor key with `work:attest` capability
5. **Nonce binding:** Verify the attestation nonce was generated after the
   worker job completed (temporal ordering)
6. **Role separation:** Verify the worker and attestor used different keys
   (different agents)

Output is a structured verification report:

```
work_01XYZ... — Cryptographic Agent Identity ADR
├─ CA: cagent-supervisor (ed25519:SHA256:abc...)
├─ worker job_01ABC... [claude/claude-sonnet-4-6]
│  ├─ certificate: VALID (expires 2026-03-18T09:00:00Z)
│  ├─ capabilities: [work:update, work:note-add]
│  ├─ commits: 3 signed, 3 verified
│  │  ├─ a1b2c3d "Add crypto identity types" — VALID
│  │  ├─ d4e5f6a "Wire capability tokens into CLI" — VALID
│  │  └─ 7g8h9i0 "Add verify command" — VALID
│  └─ updates: 2 (unsigned — metadata only)
├─ attestor job_01DEF... [codex/gpt-5.4-mini]
│  ├─ certificate: VALID (expires 2026-03-18T09:30:00Z)
│  ├─ capabilities: [work:attest, work:note-add]
│  ├─ nonce: VALID (generated after worker exit)
│  └─ attestation attest_01GHI...: PASSED — SIGNATURE VALID
└─ verdict: VERIFIED ✓
```

#### 6. Chain of Trust (choir)

In the distributed case, `cagent verify` walks a longer chain:

```
hypervisor root CA
  └─ signed VM-03 intermediate CA
       └─ signed agent-job_01ABC certificate
            └─ signed commit a1b2c3d
```

The verify command accepts `--trust-root <pubkey>` to specify which root to
verify against. For local cagent, this defaults to the supervisor CA. For choir,
it's the hypervisor root CA distributed out-of-band.

### Implementation Phases

**Phase 1: Capability tokens (CLI enforcement)**
- Generate supervisor CA keypair on first `cagent supervisor` or `cagent serve`
- Generate per-run Ed25519 keypair at dispatch time
- Sign capability tokens with CA key
- Pass token to agent via env var + temp file
- CLI checks capabilities before executing write commands
- Store agent certificate in job metadata for audit

**Phase 2: Signed git commits**
- Configure `gpg.format=ssh` and signing key per-run
- Supervisor maintains `allowed_signers` file
- `HeadCommitOID` on work items must reference a signed commit
- `cagent verify` checks commit signatures

**Phase 3: Signed attestation records**
- Add `SignerPubkey` and `Signature` fields to `AttestationRecord`
- Attestor CLI signs record before storage
- `cagent verify` checks attestation signatures

**Phase 4: Full verification tooling**
- `cagent verify <work-id>` command with structured output
- Batch verification: `cagent verify --all`
- Export verification report as JSON artifact
- choir: multi-level chain verification with `--trust-root`

### Cryptographic Choices

| Choice | Rationale |
|--------|-----------|
| Ed25519 | Fast, small keys (32 bytes), deterministic signatures, no parameter negotiation. Standard in Go (`crypto/ed25519`), SSH, and git. |
| Not JWT | JWT requires choosing algorithms, handling `alg:none` attacks, importing a JWT library. A simple `sign(canonical_json, privkey)` is auditable and has no attack surface beyond Ed25519 itself. |
| Not GPG | GPG requires a keyring daemon, web-of-trust configuration, and UID management. SSH signing uses the same Ed25519 math with zero infrastructure. |
| Not X.509 | X.509 certificates are complex (ASN.1, extensions, CRL/OCSP). Our certificates are a single signed JSON blob — human-readable, auditable, no parsing ambiguity. |
| Ephemeral keys | Agent keys live only for the duration of a run. No key rotation policy needed. Compromise of one key affects one job, not the entire system. |
| 30-minute expiry | Matches the existing lease duration. A token cannot outlive its claim. |

### Dependencies

- `crypto/ed25519` — Go standard library, no new dependencies required
- `encoding/json` — canonical JSON serialization (sorted keys, no extra whitespace)
- Git 2.34+ for SSH commit signing — available on all supported platforms

### Security Considerations

**CA key protection:** The supervisor CA private key is the root of trust. It
must be protected at rest. Phase 1 stores it as a file with mode 0600. Future
work: encrypt at rest with a passphrase or use OS keychain integration.

**Token theft:** If an agent's token file is stolen before deletion, the
attacker has a 30-minute window with scoped capabilities. Mitigation: the token
is deleted on read, and capabilities are narrow.

**Replay attacks:** Tokens include `job_id` and `expires_at`. A replayed token
for a different job_id is rejected. An expired token is rejected.

**Canonical JSON:** Signature verification requires identical serialization.
We define canonical JSON as: sorted keys, no whitespace, no trailing commas,
UTF-8 encoding. The `json.Marshal` output with sorted struct fields satisfies
this for Go structs; for maps, keys must be explicitly sorted.

### Alternatives Considered

**SPIFFE/SPIRE:** Industry-standard workload identity. Excellent for
microservices, but requires running a SPIRE server — too heavy for a local CLI
tool. choir could adopt SPIFFE SVIDs as the certificate format in the future
without changing the architecture (the hierarchy is isomorphic).

**Sigstore/cosign:** Designed for software supply chain signing with
transparency logs. Relevant for the public artifact case but requires network
access to Rekor/Fulcio. Could be layered on top for public verifiability.

**Hardware-backed keys (TPM/Secure Enclave):** Ideal for choir VMs. Out of
scope for Phase 1 but the Ed25519 interface is compatible — the private key
operations can be delegated to hardware in the future.
